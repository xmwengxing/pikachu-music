# v1.0.26 — 去掉 Render 默认后端 + 启动自动同步共享 Cookie

**日期**：2026-06-22
**目标版本**：1.0.26（versionCode 26）
**范围**：pikachu-music-rn Android App + 默认设置/启动行为
**非范围**：后端实现、扫码登录流程本身、iOS、其他平台

---

## 1. 背景与目标

当前 APK 默认带 `https://pikachu-music-api.onrender.com/api/v1` 作为 gomusic 后端地址（`src/state/settingsStore.ts:50, :134`），导致：
- 公开 APK 默认指向**维护者部署的免费 Render 实例**，所有用户共享一个后端、易触达限流/冷启动；新装用户无需自己部署就能用，但耦合维护者的资源。
- 多设备场景下，A 设备扫码登录后，**B 设备不会自动看到新 cookie** —— `App.tsx` 启动时只做了暖机 fetch，没有把后端的 cookies 写进本地 `cookieBackup`。

**本次目标**：
1. **去掉 Render 默认地址**：新装用户拿到的是干净 APK，需在设置里自行填写/部署后端地址。
2. **保留旧用户的现有设置**：已经配过 baseUrl 的升级用户不受影响。
3. **多设备共享 cookie**：App 启动 + 后端切换 + 回到前台时，自动从后端拉 cookies 合并进本地 `cookieBackup`。
4. **不做多账户**：保持现状单一 cookie 存储；不在 UI 上引入账号切换/多身份概念。

## 2. 产品决策

### 2.1 默认值变更（新装用户生效，升级用户保留旧值）

| 项目 | 旧默认值 | 新默认值 |
| --- | --- | --- |
| `gomusicBaseUrl` | `https://pikachu-music-api.onrender.com/api/v1` | `''`（空串） |
| `backends` | `[DEFAULT_RENDER_BACKEND]`（含 Render 条目） | `[]` |
| `activeBackendId` | `'render-default'` | `null` |
| `enabledSources.gomusic` | `true`（聚合源默认开） | `false`（聚合源默认关） |
| `enabledSources.migu/netease/qq/kugou/kuwo` | `true` | `true`（不变，legacy 4 平台 + 酷狗） |

**升级策略**：`STORAGE_VERSION = 3` 不变，**不写新的迁移函数**。已有用户的 `backends/activeBackendId/gomusicBaseUrl` 字段都通过 `partialize` 持久化保留。升级后他们的旧值（很可能仍指向 Render 或已切换到自建）原样存在。

唯一例外：`enabledSources` 在 `clearAll()` 时被重置为 `DEFAULT_ENABLED`。`clearAll` 是用户主动行为（设置里的"清空所有"按钮），不在升级路径里触发。

### 2.2 UI 提示

`SettingsPanel` 当 `gomusicBaseUrl.trim() === ''` 时：
- 在"后端 API"输入框下方显示一条 hint 卡片（黄色）：
  - 中文："未配置 API 地址，聚合音源不可用。如需使用请部署后端并在设置中填写地址。"
  - 英文："API URL not configured. Aggregate sources are disabled. Deploy a backend and enter its URL here."
- 卡片底部加一个"查看部署文档"链接 → `docs/GOMUSIC-API-DEPLOY.md`（在 App 内不可访问外部，可显示该路径文本，提示用户在 GitHub 看）
- "聚合 (go-music-api)"音源按钮在 baseUrl 为空时**仍然可选**（不强锁定）—— 用户先勾选再填地址是合法路径；勾选后搜索若失败，按现行逻辑回落 legacy。

### 2.3 多设备共享 cookie 行为

**当前问题**：App 启动时 `App.tsx:48-59` 用裸 `fetch('/system/cookies')` 做暖机，结果丢弃，没有写进 `settings.cookieBackup`。结果 B 设备本地缓存为空 → UI 显示"未登录"。

**新行为**：在三个时机自动从后端拉 cookies 并合并到 `cookieBackup`：

| 触发时机 | 行为 |
| --- | --- |
| App 启动后 3 秒 | 拉一次，写入 cookieBackup |
| `activeBackendId` 变化（即用户在设置里切换后端） | 立即拉一次 |
| `AppState` 变为 `'active'`（App 从后台回到前台） | 拉一次 |

**合并策略**（覆盖式 + 长者胜）：
- 后端返回 `Record<source, cookie>`，对每个 source：
  - 本地无值 → 用后端值
  - 本地有值，后端无值 → 保留本地值
  - 两者都有 → 保留**长度更长的**那条（扫码登录得到的 cookie 通常更长；用户手动粘贴的可能更短；保护用户输入不被远程覆盖）
- 同步完成后调用 `settings.loadFromObject({ cookieBackup })` 触发持久化。

**失败处理**：
- 后端不可达 / 网络错误 → 静默，不弹错（用户可能没填 baseUrl，本来就不该拉）。
- baseUrl 为空 → 整个同步流程跳过。

### 2.4 多账户

**不引入**。保持 `cookieBackup: Record<source, string>` 单层结构；不添加 `accounts[]`、账号切换 UI、多身份隔离。

**多设备 = 多个客户端连同一个后端**，后端 cookies.json 是共享存储，本次实现就是让每个客户端启动时自动看到这个共享存储。

### 3.2 新模块：`src/api/cookieSync.ts`

```ts
// 签名（具体实现细节在 implementation plan 里定）
export async function syncCookiesFromBackend(): Promise<{
  ok: boolean;
  updated: number;     // 实际写回的 cookie 条数
  reason?: string;     // 失败原因（ok=false 时）
}>;
```

内部：
1. 取 `getGomusicBase()`，空则返回 `{ ok: false, reason: 'no-base-url', updated: 0 }`
2. 调用 `listCookies()` 拿到 `Record<string, string>`
3. 读 `settings.cookieBackup`，按"长者胜"规则合并
4. 有变化 → `settings.loadFromObject({ cookieBackup: next })`
5. 返回 `{ ok: true, updated: 实际变化条数 }`

### 3.3 App.tsx 改动点

替换 `:48-69` 的两个 useEffect 为：

```ts
useEffect(() => {
  // 启动 3 秒后同步一次
  const t = setTimeout(() => { syncCookiesFromBackend().catch(() => {}); }, 3000);
  // AppState 回到 active 时同步
  const sub = AppState.addEventListener('change', (s) => {
    if (s === 'active') syncCookiesFromBackend().catch(() => {});
  });
  return () => { clearTimeout(t); sub.remove(); };
}, []);

const activeId = useSettingsStore((s) => s.activeBackendId);
useEffect(() => {
  if (!activeId) return;
  syncCookiesFromBackend().catch(() => {});
}, [activeId]);
```

## 4. 数据流（多设备共享 cookie）

```
[A 设备] 用户扫码登录 qq
   └→ POST /system/qr_login/qq_wx?key=xxx  (返回 cookie)
       └→ 后端把 cookie 写入 cookies.json (共享存储)
       └→ 后端返回 cookie 字符串给 A 客户端
           └→ A 客户端写 settings.cookieBackup.qq (本地备份)

[B 设备] 用户打开 App
   └→ App.tsx 启动 3 秒后
       └→ syncCookiesFromBackend()
           └→ GET /system/cookies → { qq: "...", ... }
           └→ 与本地 cookieBackup 合并 (长者胜)
           └→ 写回 settings.cookieBackup
           └→ AccountsTab 显示 qq 已登录 ✅
```

如果 B 设备后台运行后回到前台，同样会触发 `AppState='active'` 同步。

## 5. 错误处理

| 情况 | 行为 |
| --- | --- |
| `gomusicBaseUrl` 为空 | 跳过整个同步流程，不报错 |
| 后端不可达 / fetch 抛错 | 静默 catch；不弹 toast |
| 后端返回非对象 / 空对象 | `updated=0`，视为无变化 |
| 用户在设置页正在编辑 cookie（CookiesTab 打开中） | 仍然写回（用户主动编辑会覆盖；UI 自动重渲染） |
| 同步与扫码并发 | 扫码成功后立即调 `setCookieBackup` 写入本地；下次 `syncCookiesFromBackend` 用"长者胜"判断，不会被后端的旧 cookie 覆盖 |

## 7. 风险与权衡

| 风险 | 缓解 |
| --- | --- |
| 新装用户看到 baseUrl 空 + 聚合源未勾选，可能误以为 App 坏了 | 设置页 hint 卡片 + 部署文档链接；legacy 平台仍能用 |
| "长者胜"合并策略在极端情况下可能让旧的远程 cookie 一直赢 | 任何时候用户手动在 CookiesTab 改 → 走 `setCookieBackup`；下次同步若后端值更长会覆盖（这是预期，扫码登录产生的 cookie 最长） |
| `AppState` 监听在 Android 上偶发不触发 | 已有 `activeId` 变化的 useEffect 作为兜底 |
| 升级用户 baseUrl 仍指向 Render → 后端冷启动 15 秒 | 不在本次范围；后续可在 hint 卡片加"提示 Render 冷启动慢" |
| 多人共用一个后端 → 隐私问题（看见别人扫码的 cookie） | 后端 cookies.json 本就是共享设计；本 App 不引入多账户 = 不引入隔离 |
| 删除 Render 默认值后，v1.0.25 用户升级"什么也没变"，可能误以为版本没生效 | versionCode +1、CHANGELOG/release notes 显式说明 |
